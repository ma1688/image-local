## 0. 产品定位

本地化、单用户、桌面级的批量图像生成工作台，调用任意 OpenAI 兼容的图像生成接口（例：`http://127.0.0.1:8000/v1/images/generations`），用于批量"参考图生成 / 商品图风格化 / 贺卡礼盒改图 / 自定义流程"等场景。

参考 UI 截图核心能力：
- 顶部：待处理 / 已生成 / 已中选 / 整体进度
- 图片来源：选目录 + 手动多选；缩略图 + 文件名 + 大小 + 勾选
- 统一提示词（X/500 计数）
- 流程模板：4 套内置 + 自定义
- API 配置：地址、Key、模型（带"获取模型"）、尺寸、输出目录
- 生成控制：每张候选 1-9、失败重试开关、重试 1-5、生成中按钮、清空、取消
- 进度条：当前张数 + 候选索引 + ETA + 取消任务
- 实时日志：时间戳 + 级别图标 + 清空
- 生成结果：原图 + 候选 1/2/3 + 设为中选 + 下载全选 / 下载中选
- 历史记录：流程类型 Tab + 缩略图墙 +N + 成功/失败/耗时 + 失败原因

## 6. SSE 事件协议

`Content-Type: text/event-stream`，强制 header：`Cache-Control: no-cache`、`X-Accel-Buffering: no`。

```
id: <stream-id>
event: task.update
data: {"status":"running","total":12,"succeeded":3,"failed":0,"eta_seconds":138}

event: item.update
data: {"item_id":7,"status":"running","attempts":1}

event: candidate.ready
data: {"item_id":7,"idx":2,"thumb_url":"/api/files?path=...","file_url":"..."}

event: log
data: {"ts":"...","level":"info","message":"开始处理 img_009.jpg"}

event: task.done
data: {"status":"succeeded"}

event: ping
data: {}
```

事件总线后端 = **Redis Streams**：
- `XADD events:{taskid} MAXLEN ~ 1000 * type ... payload ...`
- SSE 端点 `XREAD BLOCK 15000 STREAMS events:{taskid} $`（无 `Last-Event-ID`）或 `STREAMS events:{taskid} <last-event-id>`（重连续推）
- `task.done` 之后再发 `retry: -1` 关闭，并对 stream 设置 24h `EXPIRE`

心跳：每 15s 发 `event: ping`。

## 7. Celery 任务拓扑

```python
# workers/tasks_jobs.py
@celery_app.task(bind=True, max_retries=0, soft_time_limit=600, time_limit=900)
def run_task(self, task_id: int): ...
    # 1) DB: status=running, started_at=now
    # 2) 加载 items；用 group(run_item.s(i.id) for i in items) 派发
    # 3) chord 回调 finalize_task(task_id)

@celery_app.task(bind=True, max_retries=0, soft_time_limit=600)
def run_item(self, item_id: int): ...
    # 0) Redis SETNX lock:item:{id} ttl=600 互斥（Celery 重投递的幂等保护）
    # 1) attempts += 1
    # 2) 重试整批清理：删除该 item 之前的所有 Candidate 行 + 文件
    # 3) httpx 调外部 API 生成 N 候选 → 落盘 → 写 DB
    # 4) 失败 + auto_retry + attempts <= retry_max → 退避后再走一遍
    # 5) 全部成功/失败 → 通过 event_bus.publish 发送 candidate.ready / item.update

@celery_app.task
def finalize_task(_results, task_id: int): ...
    # 汇总 succeeded/failed 计数 → 终态写入 → 发 task.done

@celery_app.task
def cancel_task_cleanup(task_id: int): ...
    # 收尾：取消时清理临时文件 / 关闭未完成 item
```

任务状态机（合法转换）：
```
pending  → running     # worker 接受
running  → succeeded   # 全部 item 成功
running  → failed      # 全部 item 失败 / 致命错误
running  → cancelled   # 用户取消
running  → interrupted # 进程崩溃
{succeeded,failed,cancelled,interrupted} → 终态
```

启动时扫描：将上次遗留 `running` 的任务一律置 `interrupted`。

取消：`POST /api/tasks/{id}/cancel`：
1. DB 置 `cancelled`
2. `celery_app.control.revoke(celery_uuid, terminate=True, signal='SIGTERM')`
3. 调用 `cancel_task_cleanup` 收尾

## 8. 边界与临界情况（审核结论）

### 8.1 文件系统
- 全程 `pathlib.Path`，对外用 `Path.as_posix()`
- Windows 长路径 → 必要时 `\\?\` 前缀；taskid 用 12 位短 hash
- 同秒并发命名冲突 → 目录 `{yyyymmddHHMMSSfff}_{taskid_short}`
- 路径穿越：所有用户传入路径 `Path.resolve().is_relative_to(allowed_root)`，否则 403
- 软链：`os.walk(followlinks=False)`
- 磁盘空间不足：`shutil.disk_usage` 预检；`OSError(ENOSPC)` 捕获
- 扫描超大目录：上限 1000，分页 + UI 提示
- 中文文件名：响应头 `filename*=UTF-8''<urlencoded>`
- 缩略图缓存 key：`sha1(absolute_path) + mtime + size`

### 8.2 图片来源
- 白名单：`.jpg .jpeg .png .webp .bmp .gif`
- 单图 > 20MB 默认拒绝（可调）
- EXIF 旋转：Pillow `ImageOps.exif_transpose`
- GIF 多帧：仅取首帧

### 8.3 Item 重试与幂等
- **重试整批清理**（已决策）：失败重试时先删该 item 之前所有候选（文件 + DB 行）
- 总尝试次数 = `retry_max + 1`
- 重试退避：1s / 3s / 6s
- 互斥：Redis `SETNX lock:item:{id}` ttl=600s（Celery 重投递保护）
- 部分候选成功视为整体失败（重试整批）

### 8.4 取消
- httpx client 监听 SIGTERM 主动关闭
- 已写一半的文件用 `*.tmp` 然后 `rename`，取消时清理 tmp
- 已 succeeded 的 item 候选保留可下载
- 取消请求晚于完成 → 200 + 当前终态（不报错）

### 8.5 外部 API 错误处理
- 401/403：整任务直接 failed（API Key 错），不重试
- 404（模型不存在）：整任务 failed
- 429：按 `Retry-After` 退避重试；任务级全局退避避免 worker 之间打架
- 5xx / 网络超时：重试
- 200 但解析失败：item 失败，记录 last_error 前 500 字符
- 内容审核拦截：item 失败，原因写入 `tasks.error_summary` 与 history
- httpx 配置：`Timeout(connect=10, read=180, write=30, pool=10)`，`Limits(max_connections=20, max_keepalive_connections=10)`

### 8.6 SSE 临界情况
- 中间层缓冲：`X-Accel-Buffering: no` + Vite proxy 不缓冲
- 浏览器同源 SSE 连接限制：前端单例 EventSource
- Last-Event-ID 续推：服务端用 Redis Streams 实现
- 高频候选事件抖动：前端用 `requestAnimationFrame` 批量提交 zustand 更新
- 后端进程重启 → 前端重连前先 GET 任务详情对账

### 8.7 数据库
- SQLite WAL + busy_timeout=5000
- 计数原子更新：`UPDATE ... SET succeeded = succeeded + 1`
- log_events 每任务保留 200 条
- 删除任务：`ON DELETE CASCADE` + 同步删除磁盘目录

### 8.8 安全
- API Key：`cryptography.fernet`，master key 在 `backend/.env`，首次启动自动生成
- 前端永不返回明文，编辑显示掩码
- CORS：dev 允许 `http://localhost:5173`，prod 同源
- `/api/files`：强校验白名单（outputs/ + 当次扫描根目录）
- 扫描任意目录：**允许**，但写审计日志（已决策 ⑥）
- zip 下载：`StreamingResponse` 流式写
- multipart 上传：单文件 ≤ 50MB，总数 ≤ 200，内容嗅探（不只看扩展名）

### 8.9 前端 UX 边界
- 刷新恢复：`localStorage.currentTaskId` + 启动时 GET + 重连 SSE
- 多 Tab：都订阅 SSE 不冲突，写操作以服务端为准
- 表单校验：API Profile / 模型 / 至少 1 张图 / prompt 非空
- 日志虚拟滚动 / 最多 500 条（已 align 截图"清空日志"按钮）
- 缩略图 `loading="lazy"`；原图走 lightbox
- 401 显式提示"API Key 无效"

### 8.10 任务级超时
- 默认 1800s，可在创建任务时覆写
- Celery `time_limit=900` 是单 item 上限；任务级别由 `finalize_task` 检查总耗时

## 9. 任务调度抽象（保留 asyncio 可拔插，已决策 ⑤）

```python
# services/task_service.py
class TaskService(Protocol):
    async def submit(self, task_id: int) -> str: ...   # 返回 worker uuid
    async def cancel(self, task_id: int) -> None: ...

# services/task_service_celery.py
class CeleryTaskService(TaskService):
    async def submit(self, task_id):
        result = run_task.delay(task_id)
        # 写入 tasks.celery_uuid
        return result.id

    async def cancel(self, task_id):
        uuid = await load_celery_uuid(task_id)
        celery_app.control.revoke(uuid, terminate=True, signal='SIGTERM')

# services/task_service_asyncio.py（备用）
class AsyncioTaskService(TaskService):
    """单进程内 asyncio + Semaphore 实现，便于本地无 Redis 时回退"""
```

通过 `core/settings.py` 的 `TASK_BACKEND=celery|asyncio` 切换。默认 `celery`。

## 14. 风险与回滚

| 风险 | 缓解 |
|---|---|
| docker 拉镜像慢 | 写明替代源；本地一键 `docker compose pull` |
| Celery 在 Win 原生不稳定 | 容器内用 Linux + prefork，避开 Windows 原生陷阱 |
| SSE 被代理缓冲 | 强制 header 已设置；prod 文档注明 Nginx `proxy_buffering off` |
| 进程崩溃 | 启动扫描 → 标 interrupted；UI 显式提示重跑 |
| 路径穿越 | 已加白名单测试 |
| 外部 API 超时 | httpx 显式超时 + 业务重试 + 错误透传 history |
| 整体方案不合适 | `TASK_BACKEND=asyncio` 一键回退到单进程 + asyncio 实现 |
